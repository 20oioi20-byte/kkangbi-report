# /kkangbi-ship — 배포·스모크 테스트용 스킬

## 언제 쓰나
- 코드 수정을 마치고 사용자에게 최종 파일을 전달할 때
- "반영이 안 된다"는 신고를 받았을 때 (배포 문제인지 코드 문제인지 구분)

## 필수 선행 작업
`docs/DEPLOY-CHECKLIST.md` 전체를 읽는다.

## 전달 전 체크리스트 (매번)
1. `admin.html` 수정 시: `node --check`로 JS 문법 검증 (`<script>` 태그 내용을 추출해 별도 .js 파일로 저장 후 실행)
2. `index.ts` 수정 시: 중괄호 개수 대조로 최소 구조 검증
3. 새 SQL이 필요하면 `schema_addendum_N_주제.sql`로 **새 파일** 생성 (기존 파일 덮어쓰지 않음), N은 지금까지 나온 것 중 가장 큰 번호+1
4. `present_files`로 수정된 파일 전부(admin.html 전체 / index.ts 전체 / 새 SQL) 전달 — patch 조각만 전달하지 않는다

## "반영이 안 된다" 신고 대응 순서
이 저장소는 **배포처가 2개**라서, "코드는 맞는데 안 보인다"는 신고의 8할은 배포 문제다.
1. 어느 URL에서 테스트했는지 확인 (`kkangbi-report.vercel.app` vs `report.xn--2l0b841ao7b.kr`)
2. 두 곳 다 재배포했는지 확인
3. 시크릿 새로고침(Ctrl+Shift+R)으로 캐시 우회 후 재확인 요청
4. 그래도 안 되면 페이지 소스 보기(Ctrl+U)에서 최근 추가한 함수명을 검색해 실제 서빙되는 코드 버전 확인 요청
5. 위 4단계를 다 거쳐도 안 되면 그때 코드 결함으로 간주하고 재조사

과거 실제로 이 순서를 안 거치고 코드만 계속 고치다가, 알고 보니 구버전이 서빙되고 있던 사례가 여러 번 있었다 (`docs/CHANGELOG.md` 2026-07-07~09 구간 다수).

## 배포 요청 문구 템플릿 (사용자에게 안내할 때)
> 실행 순서: **① SQL 실행 → ② Edge Function 재배포 → ③ admin.html 교체(두 배포처 모두)**
> 새로 추가/변경된 시크릿이 있으면 3단계 앞에 "Supabase Secrets 확인" 단계를 넣는다.
