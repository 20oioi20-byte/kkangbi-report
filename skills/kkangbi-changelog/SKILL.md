# /kkangbi-changelog — 변경 이력 관리용 스킬

## 언제 쓰나
- 기능/버그수정을 완료해서 사용자에게 전달하는 모든 턴의 마지막 (거의 항상)
- "이전에 뭐 했었지" 류의 히스토리 질문

## 작업 패턴
### 매 작업 완료 시 (필수, 생략 금지)
`docs/CHANGELOG.md` 최상단에 아래 형식으로 한 줄~여러 줄 추가:
```
## YYYY-MM-DD (또는 기간)
- 기능/수정 요약 한 줄. 관련 SQL 있으면 파일명 명시. `schema_addendum_N_주제.sql`
  - 하위 항목(버그수정 등)은 들여쓰기
```
- **사용자가 명시적으로 요청하지 않아도** 매번 갱신한다 (`AGENTS.md` 문서 동기화 의무 참고)
- 실제로 발견/수정한 **버그**는 원인까지 한 줄로 남긴다 (나중에 같은 실수 반복 방지 — 이 저장소에서 실제로 여러 번 도움이 됐던 습관: 예) "TO값 두 센터 모두 19로 고정돼 있던 버그 수정")

### 히스토리 질문 대응
1. `docs/CHANGELOG.md`를 최신 순으로 훑는다
2. 필요하면 `conversation_search`/`recent_chats` 도구로 실제 대화 맥락까지 찾아본다 (CHANGELOG는 요약이라 세부 맥락이 빠져있을 수 있음)

## 좋은 CHANGELOG 항목 vs 나쁜 항목
- 나쁨: "여러 가지 수정함"
- 좋음: "전체재직인원(TO대비) 산출 소스를 상담사_투입인원 → 재직인원(전체)로 정정 (사용자가 두 카드 값이 같게 나온다고 신고해서 발견)"

원인·계기까지 남기면 다음 세션(또는 새 대화창)의 에이전트가 왜 이렇게 짜여있는지 코드만 보고는 알 수 없는 맥락을 이해할 수 있다.
